#+TITLE: Frob Architecture
#+STARTUP: content

* Intent

Frob is designed to understand and modify itself while it is running. The
architecture therefore optimizes for semantic boundaries that are easy to
inspect, document, replace, and test independently.

Every application definition lives in the single =frob= package. Files
separate responsibilities; packages do not.

* Object graph

                    +-------------> active Lisp image

The central objects communicate through generic functions. Implementations
may be replaced without teaching the agent loop about HTTP, terminal escape
sequences, source editing, or subprocess protocols.

* Responsibilities

** =configuration.lisp=

Defines immutable runtime choices and paths. Configuration contains credential
locations, never credential values.

** =authentication.lisp=

Defines credential sources and OAuth refresh. Credentials enter memory only
inside the dynamic scope of a provider request. The Codex credential file is
an optional one-time, access-token-only import, not a runtime dependency on
Codex. Renewable credentials come only from Frob's own device flow.

** =device-authentication.lisp=

Owns Frob's browser-assisted device authorization, polling, PKCE exchange, and
publication into Frob's private credential store.

** =conversation.lisp=

Owns append-only portable records and their projection into provider input.
It persists each user message before network work begins and treats completed
provider output items as authoritative history.

** =provider.lisp=

Owns the replaceable model-provider protocol, Responses Lite request encoding,
HTTP transport, and SSE decoding. It emits semantic stream events and has no
terminal rendering logic.

** =tools.lisp=

Defines tool metadata, namespaces, validation, dispatch, and output. Every
tool call receives a matching result, including malformed and failed calls.

** =lisp-worker.lisp=

Owns a disposable SBCL subprocess and its readable S-expression protocol.
No =lisp.*= operation shares heap state with the active image.

** =self-tools.lisp=

Owns active-image inspection, exploratory mutation, definition identity, and
form-aware source primitives.

** =durable-mutations.lisp=

Owns checked mutation transactions, validated journal replay, and explicit Git
publication. Durable mutation objects carry explicit phases and survive source
restarts. Git and journal state reconcile across the narrow post-commit crash
window. Normal self source and commit tools cannot edit the launcher or recovery
artifacts.

** =terminal.lisp=

Owns a restrained primary-screen interface. Finalized transcript output is
written once and left in normal terminal scrollback. Frob never enters the
alternate screen and never clears scrollback. Only the bounded live region,
holding the unfinished status row, the padded prompt row, and any command
completion suggestions, may be repainted. Semantic styles resolve to basic
terminal palette colors and emphasis, so styled output follows the user's
terminal theme, and untrusted text is sanitized before trusted styling
controls are added around it.

** =agent.lisp=

Coordinates one turn at a time: persist user input, stream a response, append
completed output items, execute tools, append their results, and continue
until the provider completes without outstanding calls.

** =generations.lisp=

Owns compatible manifests and the Linux SBCL coordinator-and-saver checkpoint
backend. Only completed temporary cores are published.

** =main.lisp=

Roots logical application state for checkpoints, reconnects ephemeral resources,
projects durable records into the terminal, and wires interactive commands.

** =bin/= and =recovery/=

Contain the stable launcher boundary and minimal recovery runtime. =bootstrap=
builds that runtime into a read-only pristine core outside the repository. The
launcher restores terminal state and invokes the core; the source launcher is a
missing-core bootstrap only. Recovery reports sanitized crash and mutation
context, creates exact-commit recovery worktrees, and boots retained cores
without loading damaged active source. A failed retained core falls through to
another compatible generation.

* Durable mutation

A durable change follows one visible transaction across otherwise independent
components:

1. Append the proposed mutation to the journal.
2. Compile and install it in the active image.
3. Run the relevant checks.
4. Replace the matching complete top-level source form.
5. Commit the source change.
6. Append the durable result to the journal.

Exploratory mutation stops after installation and checking. Source remains
authoritative for clean rebuilds.

* Credentials and saved images

Provider instances retain credential-source pathnames but not access or
refresh tokens. Checkpoint code must run outside the dynamic request scope and
must reject attempts to save while credentials are dynamically bound. Tokens
never enter conversations, journals, logs, or manifests.
